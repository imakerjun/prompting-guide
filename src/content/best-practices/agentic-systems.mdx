## 장기적 추론과 상태 추적

Claude의 최신 모델은 탁월한 상태 추적 능력으로 장기적 추론 작업에 뛰어납니다. 한 번에 모든 것을 시도하기보다 **점진적 진전**에 집중하여 확장된 세션 전반에서 방향감을 유지합니다.

## 맥락 인식과 다중 윈도우 워크플로우

Claude 4.6과 4.5 모델은 **맥락 인식** 기능을 갖추고 있어, 대화 전반에 걸쳐 남은 컨텍스트 윈도우("토큰 예산")를 추적할 수 있습니다.

### 컨텍스트 제한 관리

```
당신의 컨텍스트 윈도우는 제한에 도달하면 자동으로 압축되어,
중단한 곳에서 무기한으로 계속 작업할 수 있습니다. 따라서
토큰 예산 문제로 작업을 일찍 중단하지 마세요. 토큰 예산
제한에 접근하면, 컨텍스트 윈도우가 갱신되기 전에 현재 진행
상황과 상태를 메모리에 저장하세요. 항상 가능한 한 끈기 있고
자율적으로 작업을 완전히 완료하세요. 남은 컨텍스트와 관계없이
어떤 작업도 인위적으로 일찍 중단하지 마세요.
```

### 다중 컨텍스트 윈도우 워크플로우

- **첫 번째 컨텍스트 윈도우**에서 프레임워크 설정 (테스트, 스크립트) → 이후 윈도우에서 반복
- **구조화된 형식으로 테스트 작성** 지시 (예: tests.json)
- **편의 도구 설정** 장려 (예: init.sh로 서버, 테스트, 린터 시작)
- **새로 시작 vs 압축**: Claude는 로컬 파일시스템에서 상태를 발견하는 데 매우 효과적

```
이것은 매우 긴 작업이므로, 작업을 명확하게 계획하는 것이
좋을 수 있습니다. 작업에 전체 출력 컨텍스트를 사용하는 것이
권장됩니다 — 다만, 중요한 커밋되지 않은 작업이 남아 있는
상태에서 컨텍스트가 부족하지 않도록 하세요. 이 작업을
완료할 때까지 체계적으로 계속 작업하세요.
```

### 상태 관리 모범 사례

| 방법 | 용도 |
|------|------|
| JSON 등 구조화된 형식 | 테스트 결과, 작업 상태 등 구조화된 정보 |
| 자유 형식 텍스트 | 진행 노트, 일반적 진행 상황 |
| Git | 수행 로그, 체크포인트 복원 |

## 자율성과 안전성의 균형

<Callout emoji="⚠️">
가이드 없이는 Claude Opus 4.6이 파일 삭제, 강제 푸시, 외부 서비스 게시 등 **되돌리기 어려운 작업**을 수행할 수 있습니다.
</Callout>

```
당신의 행동의 가역성과 잠재적 영향을 고려하세요.
파일 편집이나 테스트 실행 같은 로컬적이고 가역적인 행동은
자유롭게 취하도록 권장되지만, 되돌리기 어렵거나 공유
시스템에 영향을 미치거나 파괴적일 수 있는 행동에 대해서는
진행하기 전에 사용자에게 확인하세요.

확인이 필요한 행동의 예:
- 파괴적 작업: 파일/브랜치 삭제, 데이터베이스 테이블 삭제
- 되돌리기 어려운 작업: git push --force, git reset --hard
- 다른 사람에게 보이는 작업: 코드 푸시, PR/이슈 코멘트
```

## 연구와 정보 수집

```
이 정보를 구조화된 방식으로 검색하세요. 데이터를 수집하면서
여러 경쟁 가설을 개발하세요. 보정을 개선하기 위해 진행
노트에 신뢰 수준을 추적하세요. 접근 방식과 계획을 정기적으로
자기 비판하세요. 정보를 유지하고 투명성을 제공하기 위해
가설 트리 또는 연구 노트 파일을 업데이트하세요.
```

## 서브에이전트 조율

Claude의 최신 모델은 작업이 위임에 적합한 때를 인식하고 **명시적 지시 없이도** 적극적으로 수행합니다.

<Callout emoji="⚠️">
Claude Opus 4.6은 서브에이전트에 대한 **강한 선호**를 가지고 있어, 더 단순한 접근이 충분한 상황에서도 서브에이전트를 생성할 수 있습니다.
</Callout>

```
작업이 병렬로 실행될 수 있거나, 격리된 컨텍스트가 필요하거나,
상태를 공유할 필요 없는 독립적인 작업 흐름일 때 서브에이전트를
사용하세요. 간단한 작업, 순차적 작업, 단일 파일 편집, 또는
단계 간에 맥락을 유지해야 하는 작업에서는 위임하지 말고
직접 작업하세요.
```

## 복잡한 프롬프트 체이닝

적응형 사고와 서브에이전트 조율로 Claude는 대부분의 다단계 추론을 **내부적으로** 처리합니다. 명시적 프롬프트 체이닝은 중간 출력을 검사하거나 특정 파이프라인 구조를 강제해야 할 때 사용하세요.

가장 일반적인 패턴: **초안 생성 → 기준 대비 검토 → 검토 기반 개선**

## 에이전트 코딩 팁

### 파일 생성 줄이기

```
반복을 위해 임시 새 파일, 스크립트, 헬퍼 파일을 생성한 경우,
작업 종료 시 이 파일을 제거하여 정리하세요.
```

### 과열함(Overeagerness) 방지

```
과도한 엔지니어링을 피하세요. 직접 요청되거나 명확히 필요한
변경만 수행하세요. 솔루션을 단순하고 집중적으로 유지하세요.
요청된 것 이상의 기능 추가, 코드 리팩토링, "개선"을
하지 마세요.
```

### 환각 최소화

```xml
<investigate_before_answering>
열어보지 않은 코드에 대해 추측하지 마세요. 사용자가 특정
파일을 참조하면, 답변하기 전에 반드시 해당 파일을 읽어야
합니다. 코드베이스에 대한 질문에 답하기 전에 반드시 관련
파일을 조사하고 읽으세요. 근거 있고 환각 없는 답변을
제공하세요.
</investigate_before_answering>
```
