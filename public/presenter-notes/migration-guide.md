# 발표자 노트: 마이그레이션 가이드

#### 슬라이드 1: 도입부 — 모델이 바뀌면 프롬프트도 바뀐다

[오프닝 — 최종강의 시작 (2분)]

여러분, 13강의 여정이 이제 마지막이에요.

지난 시간에 기능별 팁을 다뤘고,
수정자 프레이밍이라는 실용적인 기법을 배웠죠.

오늘의 주제는 **마이그레이션 가이드**입니다.

📌 화면: 마이그레이션 관련 섹션

왜 이 주제가 마지막일까요?

AI 모델은 계속 발전합니다.
Claude 3.5가 있었고, 4.0이 나왔고, 4.5가 나왔고, 4.6이 나왔어요.
앞으로도 새로운 모델이 계속 나올 거예요.

그때마다 여러분의 프롬프트도 **조정이 필요**합니다.

비유하자면 이런 거예요.
스마트폰을 바꿀 때마다 앱 배치가 달라지잖아요.
새 폰에 맞게 설정을 다시 하는 것처럼,
새 모델에 맞게 프롬프트를 다시 조정하는 거예요.

#### 슬라이드 2: Sonnet 4.5 → Sonnet 4.6 마이그레이션

[노력 설정 — 실질적 변화 (3분)]

📌 화면: Sonnet 마이그레이션 섹션

구체적인 마이그레이션 사례를 보겠습니다.
Claude Sonnet 4.5에서 4.6으로 넘어갈 때 뭐가 달라졌을까요?

가장 큰 변화는 **기본 노력 수준**이에요.
Sonnet 4.6은 기본 노력이 **high**입니다.

그래서 대부분의 애플리케이션에서는 **medium**을 권장하고,
대량 처리나 빠른 응답이 필요한 작업에서는 **low**를 권장해요.

이건 중요한 시사점이 있어요.
새 모델이 나오면 기본 설정이 달라질 수 있다는 거예요.
예전 모델에서 잘 작동하던 프롬프트가
새 모델에서는 과하게 반응할 수 있고, 반대로 부족할 수도 있어요.

#### 슬라이드 3: 확장 사고 → 적응형 사고

[사고 방식의 진화 (3분)]

📌 화면: 확장 사고 vs 적응형 사고 코드 비교

10강에서 잠깐 다뤘던 내용인데요.
사고 방식도 마이그레이션이 필요합니다.

이전에는 **확장 사고**였어요.
"32000 토큰까지 생각해"라고 예산을 정해주는 방식이었죠.

이제는 **적응형 사고**예요.
Claude가 쿼리의 복잡도에 따라 스스로 사고량을 조절합니다.
거기에 노력 매개변수로 전체적인 수준을 설정하고요.

API를 쓰시는 분들은 코드 변경이 필요하고,
Claude.ai를 쓰시는 분들은 자연스럽게 적용되니까 걱정 안 하셔도 됩니다.

핵심은 이거예요.
**모델이 똑똑해질수록, 우리가 세세하게 제어할 필요가 줄어든다.**
예전에는 "몇 토큰까지 생각해"라고 지정했지만,
이제는 "많이 생각해" 또는 "적당히 생각해" 정도면 충분해요.

#### 슬라이드 4: 적응형 사고의 실전 활용

[워크로드별 권장 설정 (2분)]

📌 화면: 적응형 사고 워크로드 표

적응형 사고를 언제 쓰면 좋을까요?

자율 다단계 에이전트 — 코딩 에이전트, 데이터 분석, 버그 찾기 같은 작업.
노력은 **high**로 설정하세요.

컴퓨터 사용 에이전트 — 최고 수준의 정확도가 필요할 때.
역시 **high**예요.

쉬운 작업과 어려운 작업이 섞인 경우 — 이모드 워크로드라고 하는데요.
**medium에서 high** 사이로 설정합니다.

정리하면, 작업이 복잡할수록 노력을 높이고,
단순할수록 낮추면 됩니다.
상식적이죠?

#### 슬라이드 5: 전체 13강 여정 회고

[마무리 — 전체 여정 돌아보기 (3분)]

📌 화면: 전체 정리

자, 이제 전체 13강의 여정을 돌아볼게요.

**Ch.0** OT에서 뭐라고 했죠?
"공식문서를 스스로 읽는 힘을 기르자."
이게 이 강의의 출발점이었어요.

**Ch.1**에서는 "뭘 해줘"를 잘 전달하는 법을 배웠어요.

2강 — 맥락 추가하기
3강 — 예시 보여주기
4강 — XML 태그로 구조화하기
5강 — 역할 부여하기
6강 — 긴 프롬프트 다루기

**Ch.2**에서는 "결과물의 품질"을 제어하는 법을 배웠어요.
7강 — 출력과 형식
8강 — 도구 사용
9강 — 사고와 추론
10강 — 에이전트 시스템
11강 — 기능별 팁
그리고 오늘 12강 — 마이그레이션 가이드

이 모든 걸 관통하는 하나의 원칙이 있어요.

**"AI에게 명확하게, 구체적으로, 구조적으로 전달하라."**

이건 사실 사람 사이의 커뮤니케이션에서도 같은 원칙이에요.
프롬프트 엔지니어링은 결국 커뮤니케이션 스킬이에요.

#### 슬라이드 6: 최종 메시지 — 공식문서를 읽는 힘

[클로징 — OT로 회수 (2분)]

마지막으로 하고 싶은 이야기가 있어요.

오늘 배운 마이그레이션 가이드의 핵심은 뭐였죠?
**모델이 바뀌면 프롬프트도 달라져야 한다.**

그런데 모델은 앞으로도 계속 바뀝니다.
Claude 5.0이 나올 거고, 6.0이 나올 거고,
그때마다 새로운 특성이 생기겠죠.

그때 여러분에게 필요한 건 뭘까요?
**누군가 정리해줄 때까지 기다리는 게 아니라,
공식문서를 펼쳐서 "왜 이렇게 바뀌었을까?"를 스스로 추론할 수 있는 힘**이에요.

OT에서 제가 이렇게 말씀드렸거든요.
"이 강의를 다 듣고 나면
어떤 AI 모델의 공식문서든 펼쳐서
스스로 읽고 적용할 수 있게 되는 것이 목표입니다."

13강에 걸쳐서 여러분은 그 힘을 길렀어요.

명확하게 지시하는 법,
맥락과 예시를 주는 법,
구조화하는 법,
출력을 제어하는 법,
모델 변화에 대응하는 법.

이제 어떤 모델이 나와도
공식문서를 펼쳐서 읽을 수 있는 눈이 생겼습니다.

오늘의 액션, 그리고 이 강의의 마지막 액션.
앤트로픽 공식문서를 한번 직접 열어보세요.
docs.anthropic.com
이 강의에서 다뤘던 내용이 거기 다 있어요.
이제 여러분은 그걸 읽을 수 있습니다.

13강 동안 수고 많으셨습니다.
감사합니다.
