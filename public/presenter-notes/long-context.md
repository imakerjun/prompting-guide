# 발표자 노트: 긴 컨텍스트 프롬프팅

#### 슬라이드 1: 오프닝 — 공감과 문제 제기

[오프닝 — 공감과 문제 제기 (2분)]

여러분 상상해 보세요.
300페이지짜리 두꺼운 책을 한 권 받았어요.
처음부터 끝까지 쭉 읽었습니다.

그리고 누군가가 이렇게 물어요.
"5페이지에 뭐라고 써있었어?"

기억나세요?
아마 잘 기억 안 날 거예요.
295페이지를 더 읽으면서
앞부분 기억이 흐려졌을 테니까요.

그런데 만약 그 사람이 책을 주기 전에 먼저 이렇게 말했다면요.
"이 책에서 5페이지에 나오는 계약 해지 조건을 찾아줘."

그러면 어떨까요?
읽으면서 5페이지를 특별히 주의 깊게 볼 거잖아요.

이게 바로 AI한테 긴 문서를 넘길 때 일어나는 일이에요.
그리고 이게 바로 오늘 배울 핵심입니다.

여러분 업무하다 보면
AI한테 긴 문서를 넘겨야 할 때가 많잖아요.
계약서 100페이지, 분기 보고서 50페이지,
사내 규정집 200페이지.

이런 걸 그냥 복사해서 붙여넣고
"분석해줘"라고 하면 어떻게 될까요?

의외로 결과가 들쭉날쭉해요.
어떤 때는 핵심을 잘 잡는데
어떤 때는 엉뚱한 부분을 요약하거나
중요한 내용을 빠뜨리기도 해요.

AI가 긴 문서를 못 읽어서가 아닙니다.
**우리가 문서를 어떻게 배치하고 질문하느냐**에 따라
결과가 크게 달라지는 거예요.

📌 화면: 페이지 상단 "긴 컨텍스트 프롬프팅" 제목이 보이도록

#### 슬라이드 2: 가추법 읽기 — "컨텍스트(Context)" 단어 해부

[1단계: 가추법 읽기 — 단어 해부와 공식문서 (6분)]

자 오늘 주제의 단어부터 뜯어볼게요.
긴 컨텍스트 프롬프팅.

여기서 컨텍스트.
이 단어 많이 들어보셨죠?
영어로 Context.

그런데 이 단어의 어원이 재밌어요.
라틴어 contextere에서 왔는데
뜻이 뭔지 아세요?
**함께 짜다**예요. 직물을 짜는 것처럼.

즉 컨텍스트란 단순히 "정보"가 아니에요.
여러 정보가 **서로 연결되어 짜인 것**이 컨텍스트예요.

그래서 100페이지 계약서를 AI한테 넘기면
그게 하나의 커다란 직물이 되는 거예요.
실 한 가닥 한 가닥이 서로 연결되어 있는 직물.

문제는 이 직물이 너무 크면
어디가 중요한 부분인지 찾기 어렵다는 거죠.

#### 슬라이드 3: 공식문서 첫 문장 + 원칙 1 — 질문을 맨 아래에

📌 화면: 공식문서 첫 문장

자 공식문서의 첫 문장을 같이 읽어볼게요.

"대용량 문서 — 보통 A4 기준 15-20페이지 이상 — 로 작업할 때는
프롬프트를 신중하게 구조화해야 최상의 결과를 얻을 수 있습니다."

여기서 주목할 게 두 가지 있어요.

첫째, "A4 15-20페이지"라는 기준.
이게 생각보다 금방 넘어가거든요.
회의록 몇 달치, 계약서 한 건, 규정집 일부만 넣어도
이 분량은 쉽게 넘깁니다.
여러분이 업무에서 다루는 문서 대부분이 이 범위 안에 들어와요.

둘째, **"신중하게 구조화해야"**라는 표현.
앤트로픽이 왜 "구조화하면 좋다"가 아니라
"구조화해야"라고 했을까요?
Must예요. Should가 아니라.
그만큼 긴 문서에서는 구조화가
선택이 아니라 필수라는 뜻이에요.

📌 화면: "1. 긴 데이터를 상단에, 질문을 하단에 배치" 섹션

자 공식문서가 제시하는 첫 번째 원칙입니다.
**긴 데이터를 위에, 질문을 아래에.**

여기서 극적인 비유를 하나 해볼게요.

여러분이 학생이라고 생각해 보세요.
선생님이 수업 시작할 때 이렇게 말해요.
"자, 오늘 시험 문제를 먼저 알려줄게.
3번 문제는 광합성의 과정을 설명하는 거야.
그럼 이제 수업 시작하자."

이러면 어떨까요?
수업 내내 광합성 부분에 집중하겠죠.
다른 내용은 대충 듣고요.

반대로 수업을 다 듣고 나서
마지막에 시험 문제를 받으면요.
수업 전체를 이해한 상태에서 문제를 푸니까
훨씬 정확한 답을 쓸 수 있어요.

질문을 마지막에 놓는 건
마치 수업 끝에 시험 문제를 주는 것과 같아요.
Claude가 전체 문서를 먼저 다 파악한 뒤에
질문을 만나니까
"아, 이 부분이 중요하겠구나" 하고 정확히 찾아낼 수 있는 거죠.

📌 화면: 📊 Callout "쿼리를 끝에 배치하면 응답 품질이 최대 30%까지 향상" 블록

그리고 여기.
이 숫자를 같이 읽어볼게요.

"쿼리를 끝에 배치하면 테스트에서
응답 품질이 **최대 30%까지** 향상될 수 있습니다."

30%.
여러분 이 숫자가 얼마나 큰 건지 아세요?

AI 업계에서 모델 성능을 1-2% 올리기 위해
수천억 원의 연구비를 쓰거든요.
그런데 우리는 그냥 순서를 바꾸는 것만으로 30%예요.

돈 한 푼 안 들이고.
기술적 지식도 필요 없이.
그냥 질문을 아래로 옮기기만 하면 됩니다.

이건 안 할 이유가 없죠.

#### 슬라이드 4: 원칙 2 — XML 태그로 구조화

📌 화면: "2. XML 태그로 문서 콘텐츠와 메타데이터를 구조화" 섹션

두 번째 원칙.
**XML 태그로 구조화하기.**

이전에 XML 태그를 배웠죠?
긴 문서에서 XML 태그가 특히 강력한 이유가 있어요.

상상해 보세요.
서류 100장이 바닥에 흩어져 있어요.
이 중에서 A 계약서와 B 계약서를 비교해야 합니다.

서류가 섞여 있으면 어떤 게 A 계약서고
어떤 게 B 계약서인지 구분이 안 되잖아요.

그런데 각 계약서를 파일 폴더에 넣고
라벨을 붙여놓으면?
바로 찾을 수 있죠.

XML 태그가 바로 그 파일 폴더와 라벨이에요.

📌 화면: XML 태그 구조화 코드 블록 (document index="1", index="2" 예시)

여기 보면 document 태그로 감싸고
source에 파일명을 넣어줬잖아요.
이러면 Claude가 "1분기 보고서의 3페이지에 따르면..." 식으로
출처를 정확히 밝히면서 답변합니다.

태그 없이 나열하면
어디서부터 어디까지가 하나의 문서인지
Claude가 헷갈려요.

#### 슬라이드 5: 원칙 3 — 인용 먼저 요청

📌 화면: "3. 인용문으로 응답을 근거에 기반하게 만들기" 섹션

세 번째 원칙.
**먼저 인용하고 나서 답변하게 하기.**

이건 정말 강력한 테크닉이에요.

왜 효과적이냐면요.
100페이지 계약서에서 "해지 조건에 문제 있어?"라고 물으면
Claude가 100페이지 전체를 훑으면서 답변을 생성해야 하잖아요.
이때 기억이 좀 흐릿한 부분에서
자기가 지어내는 내용이 섞일 수 있어요.
이걸 할루시네이션이라고 하죠.

그런데 "먼저 관련 조항 원문을 인용해줘"라고 하면
Claude가 원문을 찾아서 꺼내놓는 과정에서
나머지 노이즈를 자연스럽게 걸러냅니다.

그리고 인용이 포함된 답변은 우리가 검증하기도 쉽죠.
"제15조 3항에 따르면..." 이렇게 나오면
원문이랑 대조해서 맞는지 바로 확인할 수 있으니까요.

이 세 가지를 한 줄로 요약하면
**"문서 먼저, 구조는 XML로, 인용 요청으로 마무리"**입니다.

#### 슬라이드 6: 예시 동작 — Before/After 비교

[2단계: 예시 동작 — Before/After 비교 (3분)]

자 예시를 볼게요.

📌 화면: "1. 긴 데이터를 상단에" 탭 비교 (❌ vs ✅)

실무 상황을 하나 그려볼게요.
여러분이 팀장이고
상사가 이렇게 말해요.

"이번 분기 보고서 50페이지 다 읽어봤어?
Q3 매출 왜 떨어졌는지 내일 아침까지 분석 좀 해줘."

자 이걸 AI한테 도움을 받으려고 합니다.

빨간 탭을 보면
"Q3 매출 하락 원인을 분석해줘"라는 질문이 위에 있고
그 아래에 50페이지 분량의 보고서가 붙어있어요.

이러면 Claude가 "매출 하락 원인"을 머릿속에 담고
50페이지를 훑어야 돼요.
마치 수업 시작할 때 시험 문제를 알려준 것처럼
관련 없는 정보에도 "이게 원인인가?" 하면서
주의가 분산됩니다.

초록 탭을 보면
50페이지 보고서가 먼저 나오고
그 아래에 질문이 있어요.

"위 보고서에서 Q3 매출 하락의 원인을 분석해줘.
주요 원인 3가지를 근거와 함께 정리해줘."

순서만 바꿨을 뿐인데
Claude가 보고서를 먼저 전체적으로 파악한 뒤
질문에 맞는 정보를 정확히 추출할 수 있게 됩니다.

그리고 마지막에 "근거와 함께"라고 했죠?
이게 바로 세 번째 원칙, 인용 요청이에요.
세 가지 원칙을 하나의 프롬프트에 녹인 거예요.

#### 슬라이드 7: 실무 적용 — 계약서, 회의록, 규정집

[3단계: 실무 적용 — 일상 업무 사례 (3분)]

📌 화면: "실무에서 이렇게 쓰세요" 섹션 스크롤

그러면 실무에서 어떻게 쓸 수 있는지 볼게요.

📌 화면: 100페이지 계약서 검토 UseCaseCard

첫 번째 사례.
여러분 상사가 이렇게 말해요.
"이 계약서에 해지 조건 문제없는지 확인 좀 해줘."

100페이지 계약서예요.
이걸 직접 다 읽으면 반나절이에요.

이때 세 가지 원칙을 적용합니다.
계약서 전문을 contract 태그로 감싸서 위에 넣고
아래에 검토 포인트를 적습니다.

"해지 조건, 지적재산권 귀속, 불리한 조항을 찾아줘.
각 항목에 대해 해당 조항 원문을 인용한 뒤 요약해줘."

문서가 위, 질문이 아래.
XML 태그로 감싸기.
인용 요청.
세 가지 원칙이 전부 들어갔죠.

두 번째. 1년치 회의록 분석.
이건 XML 태그가 특히 빛을 발하는 사례예요.

각 회의를 meeting 태그로 감싸고
date, attendees, content를 구분해 놓으면
Claude가 시간 순서와 참석자 맥락을 정확히 파악합니다.

"반복 논의됐지만 해결 안 된 이슈 Top 3"
이런 질문에 정확하게 답할 수 있는 거죠.

서류 100장을 바닥에 흩어놓은 것과
파일 폴더에 정리해놓은 것의 차이예요.

세 번째. 규정집 검색.
150페이지 인사규정에서 "육아휴직" 관련 조항을 찾을 때
"먼저 관련 조항의 원문을 정확히 인용해줘"라고 요청하면
할루시네이션, 즉 없는 내용을 지어내는 것을 크게 줄일 수 있어요.

인용을 먼저 요청하는 건
마치 변호사한테 "근거 조항을 먼저 찾아주세요"라고 하는 것과 같아요.
근거 없이 의견부터 말하는 변호사보다
조항을 먼저 찾아놓고 의견을 말하는 변호사가 더 신뢰가 가잖아요.

#### 슬라이드 8: 마무리 — 정리와 통찰

[마무리 — 정리와 통찰 (2분)]

자 이렇게 해서
긴 컨텍스트 프롬프팅을 깊이 있게 살펴봤어요.

오늘 배운 핵심을 세 가지로 정리할게요.

하나, **질문은 항상 맨 아래에.**
순서만 바꿔도 품질이 30% 올라갑니다.
수업 끝에 시험 문제를 주는 것처럼요.

둘, **여러 문서는 XML 태그로 구분.**
파일 폴더에 라벨을 붙이는 것처럼
출처 추적과 비교 분석이 정확해집니다.

셋, **"먼저 인용하고 답변해줘."**
이 한 마디가 할루시네이션을 방지합니다.
근거 없는 답변을 근거 있는 답변으로 바꿔주는 마법의 문장이에요.

그런데 여러분 한번 생각해 보세요.
이 세 가지 원칙이 AI한테만 해당될까요?

사람한테 업무를 맡길 때도 똑같잖아요.
배경 자료를 먼저 주고 질문은 마지막에.
여러 자료는 깔끔하게 정리해서.
그리고 "근거를 먼저 찾아서 보고해줘."

좋은 프롬프트는 좋은 업무 지시와 같습니다.
AI한테 잘 하면 사람한테도 잘 하게 되고
사람한테 잘 하면 AI한테도 잘 하게 돼요.

긴 문서를 AI한테 넘길 일이 많은 분들은
이 세 가지를 습관으로 만들어 보세요.
아래 연습문제에서 직접 다중 문서 비교 프롬프트를 구성해 보시고요.
